# Claude Code & Codex 5h/Weekly Usage 数据接口

本文档描述 **Claude Code** 与 **OpenAI Codex CLI** 在本地暴露 5 小时窗口与 7 天(weekly)窗口配额使用率的方式,不涉及任何具体消费方实现。任何工具都可以基于这两条路径采集数据。

---

## 总览

两个 agent 完全没有共同的数据出口,采集策略也不同:

| Agent | 数据出现位置 | 采集方式 | 触发条件 | 需要安装 |
|---|---|---|---|---|
| Claude Code | StatusLine hook 的 stdin payload | 注册一个 hook 脚本 | 每次助手回合后状态行刷新 | 是 |
| Codex CLI | rollout JSONL 的 `event_msg.token_count.rate_limits` | tail 文件 | 模型每次返回 token_count 事件 | 否 |

两个出口都是 Anthropic / OpenAI 实现细节,**非公开 API**,字段可能在版本升级时改动。建议所有解析使用宽松反序列化(默认值兜底)。

---

## 一、Claude Code

### 1.1 transcript JSONL 不携带配额

每条 `assistant` 行的 `message.usage` 只记录该轮的 `input_tokens` / `output_tokens` / `cache_*_input_tokens`,**不包含**任何窗口级累计百分比。配额值仅存在两处:

- Anthropic API 流式响应内(每次真发请求才有)
- Claude Code 转手写进 **StatusLine hook payload**

不调远程 API 时,只能利用第二个出口。

### 1.2 StatusLine hook 协议

Claude Code 在每次状态行刷新时,把一段 JSON 通过 **stdin** 喂给 `~/.claude/settings.json` 中 `statusLine.command` 指向的可执行文件,5 秒超时:

```json
{
  "session_id": "…",
  "model": { "id": "claude-opus-4-7", "display_name": "Opus 4.7" },
  "workspace": { "current_dir": "/path/to/project" },
  "rate_limits": {
    "five_hour": { "used_percentage": 35.0, "resets_at": 1774715000 },
    "seven_day": { "used_percentage": 12.0, "resets_at": 1775320000 }
  }
}
```

字段语义:

- `rate_limits.five_hour.used_percentage` — 0-100 浮点
- `rate_limits.*.resets_at` — Unix 秒
- 钩子脚本 stdout 的输出会作为状态行文本渲染

### 1.3 订阅级别影响

| 计划 | `rate_limits` 字段 |
|---|---|
| Pro / Max 订阅 | 存在,含 `five_hour` 与 `seven_day` |
| API key 用户 | **缺失** |
| Free 用户 | **缺失** |

订阅级别变化时字段可能突然消失或出现,采集端必须容忍 `rate_limits is None` 的情况。

### 1.4 settings.json 注册

```json
{
  "statusLine": {
    "type": "command",
    "command": "/absolute/path/to/your-hook.sh"
  }
}
```

**`statusLine.command` 全局只能配置一个**。多个工具想同时消费 payload 时,需要写一个 wrapper 脚本把 stdin 分发给多个下游。

### 1.5 最小钩子模板

```bash
#!/bin/bash
# Buffer stdin (5s timeout) — Claude Code may take a moment to flush
INPUT=""
while IFS= read -r -t 5 line || [ -n "$line" ]; do
    INPUT="${INPUT}${line}
"
done
[ -z "$INPUT" ] && exit 0

# Pipe via stdin (not -c arg) to avoid macOS ARG_MAX on large payloads
printf '%s' "$INPUT" | python3 -c "
import sys, json, time, os
data = json.load(sys.stdin)
rl = data.get('rate_limits')
if not rl:
    sys.exit(0)
out = {'updated_at': int(time.time())}
for k in ('five_hour', 'seven_day'):
    if rl.get(k):
        out[k] = {
            'used_percentage': rl[k].get('used_percentage', 0),
            'resets_at': rl[k].get('resets_at', 0),
        }
path = os.path.expanduser('~/cc-rate-limit.json')
tmp = path + '.tmp'
with open(tmp, 'w') as f:
    json.dump(out, f)
os.replace(tmp, path)  # atomic
" 2>/dev/null
```

实现要点:

- **stdin 用循环读 + 5s timeout**:防止 payload 异常截断时挂住
- **stdin 喂给 python 而非 `-c` 参数**:避免大 payload 触发 macOS `ARG_MAX`
- **原子写入(tmp + rename)**:避免读侧拿到截断的 JSON
- **chmod 0700**:写者与读者通常是同一用户

### 1.6 Stale 判定建议

文件长期不更新表示用户没和 Claude 交互或 Claude Code 已退出。建议消费端用 `(now - updated_at) > 10 min` 作为 stale 阈值,UI 上显示「N 分钟前」或留空,而不是直接展示旧值。

## 二、Codex CLI

### 2.1 rollout JSONL

Codex 把每轮 token 计数与配额都写进 rollout 文件:

```
~/.codex/sessions/YYYY/MM/DD/rollout-{timestamp}-{uuid}.jsonl
```

每行一条事件。abtop 等工具**无需任何安装**,只要 Codex 跑过一次,文件就在那里。

### 2.2 关键事件:`token_count`

```json
{
  "type": "event_msg",
  "timestamp": "2026-03-28T15:01:00Z",
  "payload": {
    "type": "token_count",
    "info": {
      "total_token_usage": { "input_tokens": 500, "output_tokens": 200, "cached_input_tokens": 100 },
      "last_token_usage":  { "input_tokens":  50, "output_tokens":  20, "cached_input_tokens":  10 },
      "model_context_window": 128000
    },
    "rate_limits": {
      "limit_id": "codex",
      "primary":   { "used_percent":  9.0, "window_minutes":   300, "resets_at": 1774686045 },
      "secondary": { "used_percent": 14.0, "window_minutes": 10080, "resets_at": 1775186466 },
      "plan_type": "plus"
    }
  }
}
```

字段语义:

- `total_token_usage` — 整个会话的累计计数
- `last_token_usage` — 最近一轮,适合算 context % 与画 sparkline
- `cached_input_tokens` — 旧版本字段名为 `cache_read_input_tokens`,解析时建议两者都兼容
- `rate_limits.primary` / `rate_limits.secondary` — 见下节

### 2.3 5h / 7d 槽位识别

**不要按 `primary` / `secondary` 位置硬编码**。计划差异:

| 计划 | `primary.window_minutes` | `secondary.window_minutes` |
|---|---|---|
| Plus / Pro | 300 (5h) | 10080 (7d) |
| Free | 10080 (7d) | null |

正确做法是按 `window_minutes` 分配:

```pseudo
for slot in (primary, secondary):
    if slot.window_minutes <= 300:
        five_hour ← slot
    else:
        seven_day ← slot
```

### 2.4 时间戳取值

`updated_at` 应取事件本身的 `timestamp` 字段,而非 `time::now()`。这样:

- 回放历史 jsonl 时能得到真实时间
- 同步到其他系统时不会因机器时钟漂移导致顺序错乱

### 2.5 会话发现(用于关联运行中进程)

如果工具想把 rate_limit 关联到具体的运行中 Codex 进程:

1. 用 `ps` 找命令含 `codex` 的进程(过滤 `app-server`、`grep`、`exec` 等)
2. Linux 通过 `/proc/{pid}/fd/*` 读符号链接;macOS / 其他平台用 `lsof -F pn -p{pid}`
3. 从打开的文件描述符里挑文件名匹配 `rollout-*.jsonl` 的那个

如果只想要账号级配额,不需要做这一步 —— 直接扫今天的 `~/.codex/sessions/YYYY/MM/DD/`,挑最近修改的 jsonl 即可。

### 2.6 退出后的数据持久性

Codex 退出后 jsonl 不会被删,但也不会再有新事件。希望长期展示 rate_limit 的工具应当在每次解析到更新值时**写一份本地缓存**,Codex 不在线时回退到缓存。

## 三、`RateLimitInfo` 统一模型(参考)

把两条路径产出统一成一个结构,便于多 source 并存:

```rust
pub struct RateLimitInfo {
    pub source: String,                    // "claude" | "codex"
    pub five_hour_pct: Option<f64>,
    pub five_hour_resets_at: Option<u64>,  // unix seconds
    pub seven_day_pct: Option<f64>,
    pub seven_day_resets_at: Option<u64>,
    pub updated_at: Option<u64>,
}
```

允许同一 source 出现多份(多 Claude config dir、多 Codex 计划),消费端按 `source` 分组渲染。

## 四、共同陷阱

- **Schema 不稳定** — 两端字段都是实现细节,建议所有反序列化都用「字段缺失 = 默认值」的宽松解析
- **PID reuse** — `ps` 给的 PID 可能复用,验证命令名是否仍是 `claude` / `codex`
- **没有 quota 字段不等于错误** — Free 用户、API key 用户、payload 时序异常等情况都会缺字段,正确处理为「该 source 不展示」而非崩溃
- **跨账号污染** — 多账号(个人 / enterprise)同机器使用 Codex 时,rollout 写到同一个 `~/.codex/sessions` 目录,无法仅凭文件区分账号
- **多 Claude config dir** — `CLAUDE_CONFIG_DIR` 环境变量可让 Claude Code 用非默认目录;不同会话可能用不同 config dir,要么强制单一 config,要么扫描所有候选
